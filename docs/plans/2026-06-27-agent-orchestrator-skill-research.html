Agent Orchestrator Skill Research

A research and design report for a proposed agent-orchestrator skill that lets the current agent coordinate fresh Codex CLI or Claude Code CLI worker sessions without turning the main thread into a noisy, unbounded control loop.

North Star

Create a reusable skill that helps the current agent decompose complex work, launch bounded worker sessions through Codex CLI or Claude Code CLI, monitor their progress, collect structured handoff packets, and synthesize results back in the current thread. The skill should default to conservative, inspectable orchestration: small fan-out, explicit stop conditions, least privilege, persistent logs, and clear user approval points.

Recommendation: build this as a skill first, not a plugin or daemon. Add a helper script only if you want repeatable process spawning, JSONL capture, timeout handling, and manifest creation. Do not make recursive agent spawning implicit.

Scope And Non-Goals

In scope

  • Research current orchestration, agent loop, handoff, context isolation, and parallel worker patterns.
  • Compare Codex CLI and Claude Code CLI mechanisms for creating, resuming, backgrounding, and monitoring sessions.
  • Recommend a skill architecture, default policies, resources, CLI recipes, and validation plan.
  • List decisions needed before creating the skill folder and scripts.

Out of scope for this report

  • Actually creating the agent-orchestrator skill folder.
  • Installing, repairing, or upgrading local Codex or Claude binaries.
  • Running live worker sessions against user repositories.
  • Building a persistent server, queue, database, or web UI.

Core Concepts

Agent Loop

The shared loop across current agent systems is: receive task, evaluate state, call tools or delegate, feed results back, repeat, and stop only when there are no more tool calls or a limit/approval/failure interrupts the run. This matters because orchestration is not "fire prompts at agents"; it is management of state, tool permissions, loop budgets, outputs, and failure handling.

Manager thread | |-- create bounded worker prompt |-- launch worker session |-- capture events, session id, logs, artifacts |-- wait, steer, resume, or cancel |-- require worker handoff packet | +-- synthesize final answer or next action in the main thread

Handoff Vs Manager-Style Delegation

OpenAI's orchestration guide draws a useful distinction: handoffs move conversation ownership to a specialist; agents-as-tools keep a manager in control while specialists perform bounded tasks. For this skill, the current thread should usually stay the manager. CLI-created sessions should behave like bounded workers unless the user explicitly wants a true handoff.

Pattern Use It When Skill Behavior
Manager with workers The current agent should synthesize the final result. Launch workers with explicit output contracts; collect summaries and evidence; decide centrally.
Handoff A specialist should own the next branch of work. Spawn or resume one target session with a full handoff packet and clear ownership boundary.
Native subagents The host environment already supports subagents with approval and UI integration. Prefer native Codex/Claude subagents for simple in-session fan-out; use CLI orchestration for cross-vendor, background, detached, or persistent-session needs.

Context Isolation

Codex and Claude both emphasize that long noisy tool traces degrade main-thread reliability. Worker sessions help when they keep noisy exploration, logs, failed attempts, and long file reads out of the manager context. The manager should receive only a compact final handoff packet, plus links to raw logs for audit.

Parallelism

Parallel agents work best for independent, read-heavy subtasks: codebase mapping, log analysis, review passes, documentation research, test investigation, and alternative plan generation. Write-heavy work needs stricter isolation, normally one branch or worktree per worker. Unbounded parallelism is a cost and coordination trap.

Codex Command Patterns

# Read-only research worker, JSONL capture
codex exec --json --sandbox read-only --ask-for-approval never \
  --cd /path/to/repo \
  "Research the auth module. Return a handoff packet with findings, files read, risks, and confidence."

# Write-capable worker in a controlled workspace
codex exec --json --sandbox workspace-write --ask-for-approval never \
  --cd /path/to/worktree \
  "Implement the approved change. Run targeted tests. Return files changed and evidence."

# Structured final response
codex exec "Summarize this repo" \
  --output-schema ./handoff.schema.json \
  --output-last-message ./worker-final.json

# Resume a prior non-interactive worker
codex exec resume <SESSION_ID> "Apply the reviewer feedback and rerun tests."

Claude Code CLI Surfaces

Surface Use In Skill Notes
claude -p / --print Default Claude worker mode. Runs via Agent SDK and exits. Supports piping, structured output, continuation, tool approvals, and system prompt flags.
--output-format json Preferred final capture. Returns result, session ID, usage, and cost metadata. Pair with jq and a run manifest.
--output-format stream-json --verbose Preferred event streaming. Use for real-time worker progress and tool-call visibility. Add --include-partial-messages only when token-level streaming is useful.
--bg / --background Long-running detached workers. Starts a background session and returns immediately with a short ID and management commands. Good for background investigations and multiple independent sessions.
claude agents --json Monitor background workers. Prints active sessions as JSON; --all includes completed background sessions. Interactive claude agents is useful for humans, less so for an automated skill.
claude attach/logs/respawn/rm Manage background workers. Attach to a session, print recent output, restart a session with conversation intact, or remove it from agent view while retaining transcript.
--worktree Default for write workers. Creates isolated git worktree under .claude/worktrees/<name>. Non-interactive worktrees are not auto-cleaned, so cleanup must be explicit.

Claude Command Patterns

# Read-only-ish research worker with structured result
claude --bare -p \
  --output-format json \
  --permission-mode dontAsk \
  --tools "Read,Glob,Grep" \
  "Map the auth module. Return a concise handoff packet."

# Streaming worker with logs
claude -p \
  --output-format stream-json \
  --verbose \
  --allowedTools "Read,Grep,Glob,Bash(npm test *)" \
  "Run targeted tests and explain failures."

# Background worker
claude --bg --name "flaky-test-investigation" \
  "Investigate the flaky SettingsChangeDetector test and report evidence."

# Monitor and inspect background workers
claude agents --json --all
claude logs <short-id>
claude attach <short-id>

# Continue a captured session
session_id=$(claude -p "Start review" --output-format json | jq -r '.session_id')
claude -p --resume "$session_id" --output-format json "Now focus on database queries."

Recommended Skill Architecture

Normalize the user-facing title "Agent Orchestrator" to the skill name agent-orchestrator. Keep SKILL.md lean and put longer operational details in one or two references.

File Purpose Recommendation
agent-orchestrator/SKILL.md Triggering and core workflow. Required. Include when to use, preflight, worker planning, launch, monitor, synthesize, cleanup, and safety defaults.
references/orchestration-policy.md Detailed policy and patterns. Recommended. Load when planning non-trivial fan-out, write-capable workers, handoffs, or background sessions.
references/cli-recipes.md Codex and Claude CLI command recipes. Recommended if no script is included. Keep command templates current and searchable.
scripts/run-worker.mjs Spawn workers safely and capture logs. Recommended if you want repeatability. Use Node spawn with argument arrays, never shell string interpolation. Support dry-run.
schemas/handoff.schema.json Stable worker output contract. Optional but useful. Codex supports --output-schema; Claude supports --json-schema with JSON output.
agents/openai.yaml Codex UI metadata. Recommended after the skill is drafted. Generate through the existing skill-creator script.

Proposed Trigger Description

---
name: agent-orchestrator
description: Orchestrate bounded multi-agent work by launching, monitoring, resuming, and summarizing Codex CLI or Claude Code CLI worker threads/sessions. Use when the user explicitly asks to create or coordinate agent threads, spawn CLI workers, run parallel Codex/Claude investigations, hand off work between agents, monitor background agent sessions, or synthesize results from multiple agent runs. Prefer native host subagents for simple in-session fan-out; use this skill for cross-vendor, detached, background, persistent, or CLI-managed orchestration.

Recommended Workflow

  1. Confirm intent and boundaries. Identify whether the user wants native subagents, CLI-created sessions, background tasks, or true handoff. Ask before write-capable fan-out or installing/repairing CLIs.
  2. Preflight environment. Check git status, CLI availability, versions, authentication status where safe, working directory, branch/worktree strategy, and whether logs can be written.
  3. Create a worker plan. Define each worker's objective, vendor, cwd/worktree, permissions, max runtime, output format, and stop condition. Cap default concurrency.
  4. Launch with structured capture. Prefer codex exec --json or claude -p --output-format json/stream-json. Capture stdout, stderr, exit code, session ID, command arguments, and artifact paths.
  5. Monitor and steer deliberately. Poll JSONL or claude agents --json. Steer only when the worker asks for input or the user changes direction. Cancel stuck workers after timeout.
  6. Require handoff packets. Workers must return summary, evidence, files read/changed, commands run, risks, unknowns, and recommended next action. The manager decides what enters the main thread.
  7. Integrate and verify. For write workers, inspect diffs, run tests, compare branches/worktrees, and avoid merging competing changes without a deliberate selection step.
  8. Clean up or archive. Record session IDs and log paths. Remove temporary worktrees only after preserving useful changes. Never delete transcripts or branches without explicit user direction.

Worker Handoff Packet

The handoff packet is the key anti-chaos device. Every worker prompt should demand this structure, even when the CLI output format is plain text.

{
  "worker_id": "auth-review-1",
  "vendor": "codex|claude",
  "session_id": "optional-session-id",
  "objective": "What the worker was asked to do",
  "status": "completed|blocked|failed|timed_out|needs_input",
  "summary": "Concise result",
  "evidence": [
    {
      "claim": "A concrete finding or result",
      "support": "File path, command output summary, source URL, or log path"
    }
  ],
  "files_read": [],
  "files_changed": [],
  "commands_run": [
    {
      "cmd": "npm test -- --runInBand",
      "result": "pass|fail|not_run",
      "notes": "short evidence"
    }
  ],
  "risks": [],
  "open_questions": [],
  "recommended_next_action": "What the manager should do next"
}

Safety Defaults

Permissions

Default to read-only or non-writing tool sets for research. Require explicit user intent for workspace-write, danger-full-access, Claude bypassPermissions, or broad Bash allowlists.

Concurrency

Default maximum should be small, such as 2 or 3 workers. Increase only when tasks are independent and the user accepts cost/latency tradeoffs.

Recursion

No worker may spawn further workers unless the user explicitly authorizes nested orchestration. This prevents runaway loops and unpredictable cost.

Worktrees

Use worktrees for parallel write workers. For Codex, create Git worktrees manually or use host-native worktree support where available. For Claude, prefer --worktree.

Secrets

Do not pass secrets in prompts or environment variables visible to untrusted code. Codex specifically warns against job-level API keys when repository-controlled code runs in the same environment.

Cleanup

Keep logs and manifests. Do not delete branches, transcripts, or worktrees containing changes unless the user asks. Prefer report-and-prompt for cleanup.

Risks And Mitigations

Risk Severity Mitigation
Runaway fan-out, nested agent loops, or excessive cost. High Small default concurrency, no recursive spawning by default, runtime and cost budgets, explicit worker manifest.
Conflicting writes from parallel workers. High One worktree/branch per write worker, review diffs before merging, do not let multiple workers edit same checkout.
Prompt injection through logs, web content, issue text, or worker output. High Treat worker outputs as untrusted evidence. The manager should verify file paths, commands, and claims before acting.
CLI availability and version drift. Medium Preflight versions, parse --help only as advisory, point to official docs, fail gracefully with installation guidance.
Approval prompts blocking non-interactive runs. Medium Use read-only/default modes for research; configure explicit allowlists; record blocked status and ask user instead of bypassing.
Lost context or poor handoff quality. Medium Use a required handoff schema, preserve raw logs, and ask workers for evidence not assertions.
